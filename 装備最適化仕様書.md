# RPGサーバ 装備最適化ページ 仕様書

装備の最適化ページを下記で実装してほしい。ただしビルドツールページのプログラムには触れないこと。
実装の準備：
- 装備はすべて最大強化を考慮。最大強化した装備の計算式はビルドツールの計算関数を流用。=>当然読み込む装備もビルドツールで利用しているcsvやyamlとする=>これらの設定やデータ- 式を変更すれば自動で反映されるようにすること
- 防具の叩き回数による強化やEXオプションは最適化過程で検討。忘れないように注意。特に制作武器しか叩けない等細かなゲームルール。ビルドツールのプログラムに完全準拠すれば問題ないだろう。
- 最終的な火力計算もビルドツールで出力に利用している計算関数を用いる。イメージとしてはビルドツールでユーザーが作成するビルドデータ部分のみが入力値。この入力値を最適化目標とする
フロントエンド要件：
- ユーザーが事前に定めるビルドのデータは下記
　- 職業（最大レベル固定）
　- SP振り分け。（余った分は自動探索で最適振り分けする）
　- 使用武器
　- 最適化スキル（通常攻撃、職業スキル、武器種対応スキル）
　- 必須最低値（各種ユーザーステータスのパラメータを数値入力させる。この値は最低値として満たす必要がある）
　- オプション値（最終ステータスにこの値が足される前提で最適化）
　- モード（期待値=ダメージ補正と会心率を考慮した期待値が最も大きくなる装備を探索,ステータス=任意のステータスを１つ指定させてそのステータスが最も大きくなる装備を探索,DPS=期待値にさらにスキルや武器クールタイムを考慮）
- UIやサイトデザインはセレクトメニューコンポーネントやビルドツールで利用しているSP振り分けスライドバー、デザインテーマを用いて統一感と使いやすさを意識
- 最適化の出力データは下記を折り畳み式で1～10位まで表示
　- 武器（制作武器の場合は叩き箇所と各叩き箇所の回数）
　- 防具（種類とタイプ（布、革、金属等）、叩き箇所と各叩き箇所の回数、EXステータス）
　- アクセ（種類、EXステータス）
　- 紋章（紋章の種類と、入れるストーンを各グレード４種類）
　- タロット（種類と、選択する5つのバフ（重複禁止））
- 最適化中は進捗と暫定理論装備を表示

最適化手順：
- 必須最低値（あれば）、武器種、職業バフ、スキルの4つに含まれるステータスを依存ステータスに登録。器用さもユーザの会心率が100%になるまでは依存ステータスであることに留意
- 依存ステータス内で完全上位互換が存在する装備は枝切り
- 「最適化開始」
## 詳細アルゴリズム
1) 変数を“装備そのもの”から“装備の候補バリアント”に落とす

叩き回数- EX- 紋章ストーン- タロット5枠…は、そのままだと組合せ爆発します。先にこうします：

各装備アイテムについて、ゲームルールに沿って取り得る強化パターン（叩き配分 / EX候補）を列挙

ただし全列挙は無理なので、後述の「局所的生成」にする

紋章は「石の4種×グレード」等を含むので、最初は 粗い候補集合で開始して、探索中に詳細化（後述）

この段階で各候補（バリアント）に対して

依存ステータスベクトル（依存ステのみ抽出）

必須最低値に対する寄与

（可能なら）コスト/制約メタ情報（制作武器のみ叩ける等）
を付与しておく。

ポイント：評価（火力/期待値/DPS）は最後にブラックボックスでやる。
中間では「依存ステだけ」で軽量に枝刈りする。

2) 依存ステータスでの支配関係（Pareto）による枝刈りを“スロット別”にやる

あなたの手順にある「完全上位互換」の強化版です。

スロット別（武器/胴/頭/…/アクセ/…）に候補集合を持つ

候補Aが候補Bを支配する条件（依存ステに限る）
A >= B（全次元）かつ A > B（どれか1次元） で、かつ制約メタも同等以上（例：叩き可否、条件付き効果など）

支配される候補Bは削除

さらに実務的には「近似支配」も効きます：

依存ステを丸め（例：1刻み→5刻み）て同等扱いし、候補数を抑える

3) 必須最低値を「残差ベクトル」にして探索の制約に組み込む

探索状態で持つもの：

選んだ装備の集合（途中）

現在の依存ステ合計 S

必須最低値 R に対して残り必要量 D = max(0, R - S)

（会心100%未満なら器用さを依存に含める、達成したら依存から外す or 重みを下げる）

枝刈り条件：

残りスロットで到達可能な **上界（best possible）**を事前計算しておき、

S + upperBound(残りスロット) < R ならその枝は捨てる
（必須を満たせない未来が確定）

上界は雑でOKです。例：

残り各スロットの「依存ステ合計が最大の候補」を足したもの

4) 探索本体は Beam Search（幅付き最良優先）を推奨

理由：Top10- 進捗表示- 暫定解の更新が楽。

スロット順に1つずつ装備を追加していく（例：武器→防具→アクセ→紋章→タロット）

各段階で候補を展開して、状態スコアで上位 B個だけ残す（beam幅B）

Bは可変にできる（速さ優先なら小さく、精度優先なら大きく）

状態スコア（中間の評価関数）はブラックボックスじゃなくて軽量に：

モードが「ステータス最大化」なら、そのステを重み強く

「期待値/DPS」なら、依存ステから近似スコア（例：攻撃/補正/会心寄与）を線形近似でOK
※最終順位は必ずビルドツールの火力関数で確定させる

重要：ブラックボックス評価を毎回やると重いので
“中間は近似” → “上位だけ本評価” の二段構えにする。

5) 叩き回数 / EX / タロット / 紋章は「二段階（粗→詳細）生成」にする

組合せ爆発ポイントは、最初から全部展開しないのがコツ。

例：防具の叩き

初期候補：叩き無し or 代表的な数パターン（3～10個）

Beam上位に残った状態だけ、次段で

叩き配分を局所探索（近傍：1回移す、2回移す…）

EX候補を差し替え

局所改善（hill-climb / 焼きなまし）でその状態の最良叩きを詰める

例：タロット5枠（重複禁止）

全組合せは巨大なので、まずはバフをスコア順に並べ

「貪欲 + k-best（上位k個を保持）」で候補集合を作る

上位状態だけ完全探索を少し広げる

例：紋章ストーン

まず「グレード固定で上位候補」だけ

上位状態でグレードも含めて詳細化

6) 最終評価は必ず“ビルドツールの火力関数”で行い、Top10を維持

Beamの各段階や、最終段階で、上位N状態だけブラックボックス評価

Top10はヒープで保持

同一構成（同一装備+強化）が再評価されないよう キャッシュ（hash key）を持つ

7) 進捗表示の作り方（UI要件に直結）

探索中に出すと気持ちいい情報：

現在スロット段階（例：防具まで完了）

展開数 / 残存Beam数 / 枝刈り数（上位互換- 必須未達- 上界不足）

“暫定1位”の装備構成（現時点で評価済みの中で最高）

推定残り（正確じゃなくていい）：評価済み / 予定評価数 みたいな割合
